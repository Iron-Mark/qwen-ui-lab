# Asset Replacement Audit

This audit covers visual assets in `qwen-ui-lab` and is intended to support a later AI-image replacement workflow. No replacement images were generated here.

IDs below track logical replaceable visuals. When a visual ships in several formats, the `Location` field lists every variant found so the asset is still traceable file by file.

## 1. Repository Summary

This repository is a Next.js App Router application using React, TypeScript, Tailwind CSS v4, shadcn-style UI primitives, `lucide-react` icons, Recharts, Chart.js, and PWA metadata.

Important visual asset areas:

- App routes and generated metadata: `src/app`
- Shared layout and social preview generator: `src/components/layout`
- Header/footer and shell branding: `src/features/shell/components`
- Home hero, charts, and dashboard previews: `src/features/home/components`
- Upload flow, sample picker, dynamic screenshot previews: `src/features/analysis/components`
- Sample reference registry and offline-known-sample mapping: `src/features/analysis/lib`
- Design-system social image generators and icon options: `src/features/design-system`
- Static/public assets: `public`
- App icons and PWA images: `public/icons`, `src/app/favicon.ico`
- Reference screenshots: `public/references`
- Documentation graphics: `docs/media`
- Visual regression baselines: `e2e/visual-regression.spec.ts-snapshots`

Detected asset-handling patterns:

- Static public files are referenced by absolute app paths such as `/icons/icon.svg`.
- PWA metadata is defined twice and must stay aligned: `public/manifest.json` and `src/lib/seo.ts`.
- Social preview images are generated by Next `ImageResponse` routes, not stored as image files.
- Bundled sample screenshots are registered in `src/features/analysis/lib/reference-samples.data.mjs`.
- Some SVG reference files are source material for PNG/WebP rasters generated by `scripts/generate-reference-rasters.mjs`.
- Iconography is mostly from `lucide-react`, with two inline custom social SVG marks in the footer.
- User-uploaded images, screenshot previews, detection boxes, visual-diff canvases, and chart canvases/SVGs are generated dynamically at runtime.

## 2. Asset Discovery Method

Search and inspection methods used:

- Scanned source-controlled files with `rg --files`, excluding `node_modules`, `.next`, `.git`, `test-results`, build outputs, logs, and lock/build-info files.
- Searched common image/video/vector/animation extensions: `.png`, `.jpg`, `.jpeg`, `.webp`, `.gif`, `.bmp`, `.tiff`, `.tif`, `.avif`, `.svg`, `.ico`, `.mp4`, `.webm`, `.mov`, `.m4v`, `.lottie`, `.riv`.
- Searched source, public, docs, tests, scripts, and config files for image paths, `src=`, `<Image>`, `<img>`, `url(...)`, `ImageResponse`, `opengraph-image`, `twitter-image`, `manifest`, `favicon`, `apple-touch-icon`, `maskable`, inline `<svg>`, `data:image`, and remote `http(s)` URLs.
- Inspected `public/manifest.json`, `src/lib/seo.ts`, `public/sw.js`, and `public/offline.html` for PWA and metadata asset references.
- Parsed `lucide-react` imports and custom inline SVG components.
- Extracted image dimensions and file sizes using the bundled Python/Pillow runtime and ICO header parsing.
- Rendered or visually inspected raster assets, SVG previews, and contact sheets. Two source SVGs contain invalid control characters but their PNG/WebP rasters were visually inspected.
- Used three read-only subagents as cross-checks: repository mapping, asset detection, and QA/missed-reference review.

No remote image URLs, videos, Lottie files, or Rive files were found. Remote URLs in the repository are docs, APIs, deployment URLs, social links, or schema references, not image assets.

## 6. Consolidated AI Art Direction

Overall style:

- Premium developer-tool SaaS, clean and utilitarian rather than decorative.
- UI-first visuals that look like real product screenshots and app assets.
- Clear geometry, restrained accents, strong spacing, and high information clarity.

Recommended color palette:

- Base light: near-white `#FAFAFA`, white cards, zinc/neutral text.
- Base dark: near-black navy/charcoal `#0B1020` or `#111827`.
- Primary accent: cyan `#22D3EE` or mint `#10B981`.
- Secondary accent: soft violet `#A78BFA`.
- Status accents: green for positive, red for negative, amber for warning, but used sparingly.

Composition rules:

- Screenshots should show complete viewports, not cropped fragments.
- Keep UI elements aligned to a grid.
- Leave clear safe areas for social-preview text and icon crops.
- Avoid decorative background objects that compete with UI.
- For PWA icons, keep the mark legible at small sizes.

Lighting and mood:

- Clean product-lighting feel, not photorealistic studio lighting.
- Subtle depth through shadows, borders, and soft gradients.
- Calm, reliable, engineer-friendly tone.

Texture and detail:

- Crisp vector-like UI details.
- Low-noise backgrounds.
- No photorealistic people or stock imagery.

Icon style:

- Single system, 24x24 SVG-ready, 2 px stroke, rounded caps.
- Use filled app-logo mark only for brand assets; use outline icons in UI.

Illustration style:

- If illustrations are introduced, keep them abstract and product/UI-related: panels, detection boxes, code snippets, grid overlays.
- Do not imitate living artists or existing brand systems.

Background style:

- Use subtle technical grids, UI frames, and gradients.
- Avoid orbs, blobs, bokeh, and purely atmospheric backgrounds.

Brand consistency guidance:

- Pick one central metaphor: screenshot frame + analysis lens + code/export bars.
- Use it across app icon, social cards, and any hero/marketing assets.
- Keep `qwen-ui-lab` text as editable UI or code text where possible; review third-party brand/legal implications before baking it into images.

Assets to generate as matching sets:

- A001-A005 app icon family.
- A007-A014 sample reference screenshot suite.
- A021-A024 social preview cards.
- A025-A027 icon/social mark system if moving away from current icons.

## 9. Open Questions or Risks

- The app-icon family is visually inconsistent today. Replace A001-A005 together, not one file at a time.
- `public/manifest.json` and `src/lib/seo.ts` both define PWA icon/screenshot metadata. They can drift if only one is updated.
- `public/sw.js` precaches icons, manifests, and social routes. Bump `CACHE_NAME` when replacing cached assets.
- Reference samples are filename-coupled. `reference-samples.data.mjs`, `offline-analyze.mjs`, tests, docs, and raster generation all expect known stems like `dashboard-reference`.
- `offline-analyze.mjs` contains known-sample signatures and mappings. Replacing sample visuals without updating this logic can make known sample recognition incorrect.
- `scripts/generate-reference-rasters.mjs` hardcodes reference stems and dimensions. New aspect ratios need script, metadata, manifest, and tests updated.
- `landing-reference.svg` and `ecommerce-reference.svg` contain invalid/control-character text issues. Their rasters render, but replacement should clean the SVG sources.
- Static social fallback `public/images/og-image.png` was removed after generated social preview routes were confirmed as the active metadata path.
- Starter leftovers `public/next.svg`, `public/vercel.svg`, `public/globe.svg`, `public/file.svg`, and `public/window.svg` were removed after reference search confirmed no runtime usage.
- Footer GitHub and LinkedIn inline SVG marks may require brand/legal review if kept as official social marks.
- Generated social images are not static files. Replacing them may require code changes, not image uploads.
- Chart and upload preview visuals are runtime-generated; replacing them with AI images may reduce responsiveness and accessibility.
- Visual regression snapshots are test artifacts. They should be regenerated from the app after replacement rather than manually image-generated.
- Documentation and slide files reference some old visual assets and may need updates after replacements.
